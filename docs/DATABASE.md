# DATABASE\.md

## **文档目的**

本文档用于将《社交媒体焦虑分析工具 PRD（十次修订版）》中已经确认的产品需求，以及 `ARCHITECTURE.md` 中的技术架构参考，转换为开发可执行的数据库设计说明。

本文档的作用是：

- 将 PRD 中的业务对象、页面状态、用户数据边界和权限规则转换为数据库结构；

- 明确第一版朋友内测完整体验版需要的数据表、字段、索引、关系和删除规则；

- 明确用户私人数据、公共内容、管理员数据、用户主动提交反馈之间的边界；

- 支撑登录 / 注册、截图分析、AI 结果保存、高风险提醒、历史记录、反馈、百宝箱、个人笔记、管理员后台等核心流程；

- 为后续生成 SQL 建表语句、API\.md、TASKS\.md、TEST\_CASES\.md 提供数据库依据。

本文档不替代 PRD，不反向修改 PRD，不重新审查 PRD，不删减、弱化或重新解释 PRD 中已经确认的产品需求。

`ARCHITECTURE.md` 仅作为技术架构参考。
 当某些字段或结构是数据库设计需要、但 PRD 未明确写出时，本文档会明确标记为 **「数据库设计建议」**。

## **数据库设计原则**

### **2\.1 数据库选型**

第一版朋友内测完整体验版优先使用：

- MariaDB / MySQL；

- InnoDB 存储引擎；

- `utf8mb4` 字符集；

- 支持中文、Emoji、长文本和多语言内容；

- 单机数据库部署，后续再考虑拆分。

第一版不引入：

- 分库分表；

- 独立数据仓库；

- 复杂微服务数据库拆分；

- 强依赖独立向量数据库；

- 大型消息队列；

- 复杂实时同步系统。

### **2\.2 用户数据隔离原则**

所有用户私人数据必须按 `user_id` 或 `owner_user_id` 隔离。

以下数据均属于用户私人数据：

- 历史记录；

- 原截图；

- OCR 完整识别文字；

- 手动补充关键文字；

- 用户感受；

- 分析前焦虑强度；

- 分析后焦虑强度；

- 截图来源平台；

- AI 分析结果；

- 关键词；

- 人设类型；

- 焦虑类型；

- 反馈；

- 后续感受；

- 用户备注；

- 用户自定义标签；

- 来源 / 参考依据快照；

- 免责声明快照；

- 高风险状态；

- 历史摘要；

- 个人笔记；

- 个人笔记标签；

- 头像；

- 账号凭证；

- 用户偏好；

- 搜索日志，若第一版启用。

所有用户私人数据查询必须带当前登录用户的 `user_id` 条件。

### **2\.3 管理员权限原则**

管理员不能访问用户私人分析内容。

管理员可以管理：

- 百宝箱知识库；

- 百宝箱分类；

- 百宝箱标签；

- 百宝箱首页推荐；

- 首页名言；

- 首页好歌推荐；

- 轻柔情绪问候；

- Coming Soon 文案；

- 产品说明；

- 免责声明；

- 新手教程内容；

- 用户主动提交的提议 / 反馈；

- 用户提议 / 反馈处理状态。

管理员不能查看：

- 用户历史记录；

- 用户截图；

- 用户感受；

- 用户 OCR 识别文字；

- 用户手动补充关键文字；

- 用户 AI 分析结果；

- 用户分析反馈记录；

- 用户后续感受；

- 用户高风险状态；

- 用户个人笔记；

- 用户备注；

- 用户标签；

- 用户私人搜索记录。

非管理员访问后台路由或后台 API 必须返回 403。

### **2\.4 删除与注销原则**

PRD 要求历史删除和账号注销涉及用户私人数据时采用物理删除。

因此：

- 删除历史记录为物理删除；

- 删除历史记录必须删除该历史相关截图、识别文字、感受、分析结果、反馈、后续感受、备注、标签关联、来源快照、免责声明快照和高风险标记；

- 删除历史记录不删除个人笔记；

- 若个人笔记关联被删除的历史记录，需要解除关联或保留标题快照并显示“关联来源已删除”；

- 删除个人笔记只删除该笔记，不影响历史记录或百宝箱内容；

- 注销账号必须物理删除用户私人数据；

- 用户主动提交的提议 / 反馈在注销后按 PRD 匿名化规则处理；

- 用户私人数据表不默认依赖软删除；

- 公共内容表可保留 `deleted_at`，作为 **数据库设计建议**。

### **2\.5 来源快照与免责声明快照原则**

以下内容必须在分析完成时保存快照：

- 来源 / 参考依据快照；

- 免责声明快照。

原因：

- 后续知识条目被编辑或停用，不影响已有历史详情页；

- 后续管理员修改免责声明，不影响已有历史详情页；

- 历史详情应展示分析当时使用的依据和免责声明。

### **2\.6 高风险提醒持久化原则**

高风险标记和提醒状态需要持久化。

高风险提醒至少支持三态：

- 未触发；

- 待处理；

- 已处理。

首次分析完成后若识别为高风险：

- 先生成完整分析结果；

- 先创建历史记录；

- 写入高风险标记；

- 提醒状态设为待处理；

- 结果页弹出全屏提醒遮挡结果。

如果用户未处理提醒并异常退出，再次从历史详情页打开该记录时仍触发一次。

用户选择“我已了解，继续查看分析”或“返回首页”后，状态改为已处理，后续不重复弹出。

## **通用字段规范**

### **5\.1 **

### **`id`**

所有主要业务表建议包含 `id` 作为主键。

**数据库设计建议：**

- 使用 `BIGINT UNSIGNED AUTO_INCREMENT`；

- 第一版不强制使用 UUID；

- 用户可见 URL 不直接依赖自增 ID 时，可额外使用 `record_public_id` 或 `result_url_token`。

### **5\.2 **

### **`created_at`**

记录创建时间。

适用范围：

- 所有核心表；

- 所有用户私人数据表；

- 所有公共内容表；

- 所有关系表。

### **5\.3 **

### **`updated_at`**

记录最后更新时间。

适用范围：

- 可编辑业务表必须包含；

- 纯关系表可选；

- 用户私人数据表建议保留，便于排查状态。

### **5\.4 **

### **`deleted_at`**

PRD 要求历史删除和注销为物理删除，因此：

- 用户私人数据表不要默认依赖 `deleted_at`；

- 删除历史记录时直接物理删除；

- 注销账号时直接物理删除私人数据；

- 公共内容表可保留 `deleted_at`，作为 **数据库设计建议**，用于后台内容误删恢复或隐藏。

适合保留 `deleted_at` 的表：

- `knowledge_categories`；

- `knowledge_items`；

- `admin_contents`；

- `home_quotes`；

- `home_songs`。

不建议依赖 `deleted_at` 的表：

- `analysis_records`；

- `analysis_results`；

- `analysis_feedbacks`；

- `analysis_sources_snapshot`；

- `analysis_disclaimer_snapshots`；

- `analysis_risk_states`；

- `analysis_history_summaries`；

- `notes`；

- `file_assets`；

- `user_credentials`。

### **5\.5 **

### **`status`**

用于表示业务处理状态。

常见表：

- `analysis_records.status`；

- `analysis_risk_states.risk_status`；

- `user_suggestions.status`；

- `file_assets.status`，可选；

- `admin_contents.is_enabled` 更适合用布尔字段。

状态字段应使用有限枚举，不允许任意文本。

### **5\.6 **

### **`user_id`**

所有用户私人数据表必须直接或间接绑定用户。

必须直接包含 `user_id` 的表：

- `user_credentials`；

- `user_preferences`；

- `user_profiles`；

- `user_compliance_confirmations`；

- `analysis_records`；

- `analysis_feedbacks`；

- `analysis_history_summaries`；

- `analysis_risk_states`；

- `notes`；

- `note_tags`；

- `record_custom_tags`；

- `user_suggestions`；

- `search_logs`。

通过主表间接归属用户的表：

- `analysis_results`；

- `analysis_keywords`；

- `analysis_persona_tags`；

- `analysis_anxiety_tags`；

- `analysis_sources_snapshot`；

- `analysis_disclaimer_snapshots`；

- `record_custom_tag_relations`；

- `note_tag_relations`；

- `note_source_relations`。

查询间接归属表时，必须通过主表校验 `user_id`。

### **5\.7 **

### **`sort_order`**

用于展示排序。

适用表：

- `analysis_keywords`；

- `analysis_persona_tags`；

- `analysis_anxiety_tags`；

- `analysis_sources_snapshot`；

- `knowledge_categories`；

- `knowledge_item_tags`；

- `home_quotes`；

- `home_songs`。

### **5\.8 **

### **`is_enabled`**

用于公共内容启用 / 停用。

适用表：

- `knowledge_categories`；

- `knowledge_items`；

- `admin_contents`；

- `home_quotes`；

- `home_songs`。

用户私人数据表一般不使用 `is_enabled`。

## **用户与账号相关表**

## **6\.1 **

## **`users`**

### **用途**

存储用户基础信息。

### **字段设计**

### **设计说明**

- 用户名可重复，仅作为展示字段。

- 用户名支持中文、英文、数字和下划线。

- 用户名不允许纯空格。

- 登录唯一性不由 `username` 保证，而由 `user_credentials` 保证。

- 未上传头像时，前端可根据用户名首字母生成默认彩色头像。

- 注销账号时，用户账号和账号关联数据需要物理删除。

- 注销后，用户主动提交的 `user_suggestions` 按匿名化规则保留有限字段。

---

## **6\.2 **

## **`user_credentials`**

### **用途**

存储手机号 / 邮箱登录凭证。

### **字段设计**

### **唯一索引**

### **设计说明**

- 手机号 / 邮箱作为登录凭证。

- 用户可以先绑定手机号或邮箱其中一个，后续补充绑定另一个。

- 两个已绑定凭证均可用于登录。

- 第一版不支持解绑主凭证或更换已绑定凭证。

- 密码至少 8 位，包含字母和数字。

- 修改密码时，新密码不能与旧密码相同。

- 不保存明文密码。

- `credential_value` 应保存规范化后的手机号或邮箱。

---

## **6\.3 **

## **`verification_codes`**

### **用途**

用于注册、登录验证、找回密码、绑定第二凭证等场景。

### **字段设计**

### **设计说明**

- 验证码有效期为 5 分钟。

- 重发冷却时间为 60 秒。

- 不保存明文验证码，只保存哈希。

- 验证成功后写入 `consumed_at`。

- 注册验证码按凭证类型发送：

- 



- 手机号使用短信验证码；

- 邮箱使用邮件验证码。

- 找回密码仅支持通过已绑定手机号或邮箱。

- **数据库设计建议：** 定期清理过期验证码，避免表膨胀。

---

## **6\.4 **

## **`user_preferences`**

### **用途**

存储用户偏好设置。

### **字段设计**

### **设计说明**

- 与用户一对一。

- `reference_history_default` 只影响未来新发起的分析。

- 用户在上传页临时开启或关闭参考历史时，以 `analysis_records.reference_history_enabled` 为准。

- 当用户没有历史记录时，上传页参考历史选项禁用。

- 首次引导按账号维度记录，换设备登录后不重复自动展示。

- 新手教程第一次进入首页后自动弹出一次，完成后不再自动展示。

---

## **6\.5 **

## **`user_profiles`**

### **用途**

存储用户资料扩展信息。

### **字段设计**

### **设计说明**

- 第一版如无复杂资料扩展，可不创建本表，将必要字段放入 `users`。

- 如保留本表，必须按 `user_id` 隔离。

- 注销账号时物理删除。

---

## **6\.6 **

## **`user_compliance_confirmations`**

### **用途**

记录用户对用户协议、隐私政策、免责声明的确认状态。

### **字段设计**

### **设计说明**

- 注册时必须勾选同意用户协议和隐私政策。

- 注册时必须确认已阅读免责声明。

- 跳过首次引导不等于跳过免责声明。

- 用户协议页和隐私政策页第一版为静态页面，代码维护。

- 免责声明由后台系统文案维护。

- **数据库设计建议：** 记录 `content_version`，便于未来协议或免责声明更新时追踪用户确认版本。

# **文件与附件表**

## **7\.1 **

## **`file_assets`**

### **用途**

用于存储头像、原截图等文件资产的数据库引用。

### **字段设计**

### **设计说明**

- 用户默认不保存原截图。

- 用户在结果页勾选保存原截图后，才将原截图作为历史附件持久保存。

- 结果页“保存原截图”默认不勾选。

- 删除历史时，同步删除关联截图文件和 `file_assets` 记录。

- 注销账号时，删除用户头像和私人附件。

- 文件访问必须校验 `owner_user_id = current_user.id`。

- 管理员不能通过后台访问用户截图文件。

- 上传目录不应直接作为公开静态目录暴露。

- 文件名不使用用户原始文件名，避免泄露信息。

- **数据库设计建议：** 第一版优先本地存储，预留 `storage_type` 方便后续切换对象存储。

# **分析历史相关表**

## **8\.1 **

## **`analysis_records`**

### **用途**

分析历史主表。

AI 分析成功生成结果时，自动创建历史记录。

### **字段设计**

### **设计说明**

- 分析成功后自动创建历史记录。

- 结果页 URL 包含历史记录 ID 或 `record_public_id`，仅当前登录用户有效。

- 不生成公开分享链接。

- 用户刷新结果页时从历史记录恢复，不重新触发分析。

- 访问非本人记录必须返回 403 或重定向至首页。

- 删除历史为物理删除。

- 用户感受生成结果后不可修改，若需补充说明通过备注完成。

- 备注与 AI 结论分区展示，不覆盖、不替换 AI 原文。

- 手动补充关键文字与完整 OCR 识别文字分开保存。

- 即使用户未保存完整 OCR 文字，只要用户曾手动补充关键文字，历史详情仍展示该补充文字。

- 完整 OCR 识别文字默认不保存，只有用户主动勾选后才保存。

- 原截图默认不保存，只有用户主动勾选后才关联 `screenshot_file_id`。

- **数据库设计建议：** `parent_record_id` 用于记录“基于这条记录重新分析”的来源关系，但不影响旧记录内容。

## **8\.2 **

## **`analysis_results`**

### **用途**

存储结构化 AI 分析结果。

### **字段设计**

### **设计说明**

- 13 个结果页模块中，11 个 AI 生成模块主要存储在本表及相关标签表中。

- 相关百宝箱推荐是系统推荐模块，不作为 AI 直接输出。

- 免责声明是系统渲染模块，不属于 AI 输出，不存储在本表。

- 来源 / 参考依据虽然属于结果页展示模块，但需要存入 `analysis_sources_snapshot`。

- 所有分析模块都应出现；信息不足时保存“当前信息不足，无法判断”或对应兜底文案，而不是缺失模块。

- `raw_ai_response_json` 仅用于开发排查和兜底，不直接暴露给普通用户。

---

## **8\.3 **

## **`analysis_keywords`**

### **用途**

存储每条分析历史的关键词。

### **字段设计**

### **设计说明**

- 关键词允许 1\-8 个。

- 关键词由 AI 基于截图内容、用户感受和分析结果生成。

- 关键词不强制与 OCR 提取文字完全一致。

- 若无法提取任何关键词，结果页显示“暂无关键词”。

- 关键词用于历史详情展示和历史搜索。

- 关键词不等同于用户自定义历史标签。

---

## **8\.4 **

## **`analysis_persona_tags`**

### **用途**

存储 AI 识别出的人设 / 人格化形象类型标签。

### **字段设计**

### **设计说明**

- AI 可输出多个相关人设标签。

- 结果页展示全部识别到的人设标签。

- 历史列表展示主标签。

- 当无法可靠判断时，不强行贴标签，显示“当前信息不足，无法明确识别人设类型”。

---

## **8\.5 **

## **`analysis_anxiety_tags`**

# **高风险提醒表**

## **10\.1 **

## **`analysis_risk_states`**

### **用途**

持久化每条历史记录的高风险识别结果与提醒处理状态。

### **字段设计**

### **设计说明**

- 高风险提醒三态：

- 



- `none`：未触发；

- `pending`：待处理；

- `handled`：已处理。

- 出现明确自伤、自杀、伤害自己、不想活、想消失、极端绝望等表达时触发。

- 用户焦虑强度 ≥ 8 且同时出现明显危险表达时触发。

- AI 判断不确定但风险较高时，优先触发提醒。

- 系统先完成 AI 分析、生成结果并创建历史记录，再弹出高风险提醒。

- 用户未处理提醒并异常退出时，再次从历史详情打开仍触发一次。

- 用户处理后，更新为 `handled`。

- 用户选择返回首页后，分析结果和历史记录仍保留。

- 高风险标记随历史记录删除而删除。

- 管理员不能查询本表。

- **数据库设计建议：** 第一版也可将这些字段直接放在 `analysis_records` 中，但推荐独立表，便于状态流转和权限隔离。

# **数据权限设计**

## **20\.1 用户私人数据权限**

所有用户私人数据查询必须带 `user_id` 条件。

典型规则：

- 查询历史列表：`analysis_records.user_id = current_user.id`

- 查询历史详情：先校验 `analysis_records.user_id = current_user.id`

- 查询反馈：通过 `record_id` 反查历史归属，或直接校验 `analysis_feedbacks.user_id = current_user.id`

- 查询个人笔记：`notes.user_id = current_user.id`

- 查询笔记标签：`note_tags.user_id = current_user.id`

- 查询历史自定义标签：`record_custom_tags.user_id = current_user.id`

- 查询文件：`file_assets.owner_user_id = current_user.id`

## **20\.2 历史详情权限**

历史详情必须校验：

```Plain Text
analysis_records.user_id = current_user.id
```

如果不匹配：

- 返回 403；

- 或重定向至首页；

- 不展示任何历史内容。

不得通过返回内容暴露他人记录是否存在。

## **20\.3 文件访问权限**

文件访问必须校验：

```Plain Text
file_assets.owner_user_id = current_user.id
```

用户不能通过文件 ID 或路径访问他人截图或头像。

推荐方式：

- 上传目录不放在公开静态目录；

- 用户请求图片时通过受控 API 访问；

- API 校验当前用户是否拥有该 `file_assets`；

- 校验通过后返回文件流或临时访问地址。

## **20\.4 管理员权限**

管理员身份通过 `admin_users` 判断。

管理员可以访问：

- `knowledge_categories`；

- `knowledge_items`；

- `knowledge_tags`；

- `knowledge_item_tags`；

- `home_quotes`；

- `home_songs`；

- `admin_contents`；

- `user_suggestions`。

管理员不能访问：

- `analysis_records`；

- `analysis_results`；

- `analysis_feedbacks`；

- `analysis_sources_snapshot`；

- `analysis_disclaimer_snapshots`；

- `analysis_risk_states`；

- `analysis_history_summaries`；

- `notes`；

- `note_tags`；

- `record_custom_tags`；

- 用户截图类 `file_assets`。

非管理员访问后台路由或后台 API 返回 403。

## **20\.5 用户提议 / 反馈权限**

用户主动提交的 `user_suggestions` 可供管理员后台查看和处理。

管理员可查看：

- 提议类型；

- 提交内容；

- 用户可选联系方式；

- 提交时间；

- 处理状态。

注销后必须匿名化：

- 不显示原用户；

- 不显示联系方式；

- 不保留正文；

- 后台显示“已注销用户提交”。

# **备份与恢复建议**

## **22\.1 数据库备份**

第一版至少保留开发者级备份策略。

建议：

- 每日自动备份数据库；

- 重要版本发布前手动备份；

- 备份文件加密保存；

- 定期验证备份可恢复；

- 备份权限仅开发者或项目创建者可访问。

## **22\.2 上传文件备份**

需要备份：

- 用户头像；

- 用户主动保存的原截图；

- 其他私人附件。

如果使用本地存储：

- 需要定期备份上传目录；

- 需要保证数据库 `storage_path` 与文件实际路径一致。

如果使用对象存储：

- 需要开启对象存储备份或版本策略；

- 删除历史和注销时仍必须执行物理删除。

## **22\.3 文件与数据库一致性检查**

建议定期检查：

- `file_assets.storage_path` 是否存在对应文件；

- 数据库已删除的文件是否仍残留在存储目录；

- 历史记录删除后截图是否同步删除；

- 注销用户后头像和私人附件是否同步删除。

## **22\.4 删除恢复原则**

由于 PRD 要求：

- 删除历史为物理删除；

- 注销账号为物理删除私人数据。

因此：

- 前台不向用户提供撤销恢复；

- 恢复只能依赖开发者备份；

- 如从备份恢复，应谨慎避免恢复已注销用户的私人数据；

- 删除历史和注销账号后的恢复不作为第一版用户功能。

# **数据库设计结论**

本数据库设计支持《社交媒体焦虑分析工具 PRD（十次修订版）》定义的 V0\.1 朋友内测完整体验版。

核心设计重点包括：

- 使用 MariaDB / MySQL、InnoDB、`utf8mb4`；

- 用户数据按 `user_id` 严格隔离；

- 管理员不能访问用户私人分析内容；

- 登录凭证与用户展示名分离；

- 用户协议、隐私政策、免责声明确认状态可持久化；

- 分析历史自动保存；

- 原截图默认不保存，用户主动勾选后才作为历史附件保留；

- 完整 OCR 识别文字默认不保存，用户主动勾选后才保存；

- 手动补充关键文字与完整 OCR 识别文字分开保存；

- 结构化 AI 分析结果、关键词、人设标签、焦虑标签完整保存；

- 来源 / 参考依据保存快照；

- 免责声明保存快照；

- 高风险提醒状态持久化；

- 分析反馈为用户私人记录，管理员不可见；

- 用户提议 / 反馈为主动提交给管理员的处理对象，注销后匿名化；

- 历史记录删除采用物理删除；

- 注销账号采用私人数据物理删除；

- 删除历史不删除个人笔记，只解除关联；

- 个人笔记可关联历史记录和百宝箱内容；

- 历史自定义标签、AI 关键词、百宝箱标签、个人笔记标签保持清晰边界；

- 后台仅管理公共内容和用户主动提交的提议 / 反馈；

- 第一版保持数据库结构完整但不过度平台化。
